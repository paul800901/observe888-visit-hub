# Observe888 Visit Hub / ObserveGeoPages 專案脈絡

最後更新：2026-05-10

## 一句話定位

`observe888-visit-hub` 是見觀結構的手機格式微縮官網 / 到店導航站；`ObserveGeoPages` 是另一個品牌官網與 GEO / SEO 公開文章庫專案，兩者互相連結但不互相取代。

## 為什麼會分成兩個官網

見觀結構原本已有舊官網，但舊官網是網路公司製作的模板型網站，修改權與更新速度受限，不適合承載快速迭代的內容策略、GEO / SEO 文章、AI 搜尋可讀內容與細緻追蹤。

現在新的方向是把內容發布與修改控制權收回自己手上，用 GitHub Pages、版本控管與 AI 輔助開發建立可長期維護的公開站群。

## 兩個官網的分工

### 官網一：見觀微縮官網 / 到店導航站

- 顯示名稱：見觀結構
- 專案 / repo：`observe888-visit-hub`
- 來源關係：AdsControl 內的 `observe888-visit-hub` 是微縮官網來源 / 輸出區；本 repo 是公開 GitHub Pages bundle。
- 核心用途：手機使用者快速知道要去哪裡、怎麼預約、北區 / 南區怎麼選。
- 主要功能：北區店入口、南區工作室入口、LINE 預約、電話、pricing、地圖與到店導航。
- 設計原則：微縮官網是專為手機格式做的獨立導航官網，不是舊官網的縮小版。

### 官網二：品牌官網 / GEO SEO 公開文章庫

- 專案名稱：`ObserveGeoPages`
- 建議 repo 名：`observe-geo-pages`
- 對外定位：見觀結構｜身體觀察筆記
- 核心用途：承載品牌理解、身體觀察文章、主題群、長尾搜尋、AI 搜尋可讀內容。
- 技術方向：GitHub Pages + Markdown / 靜態網站產生器 + sitemap + canonical + structured data。
- 與微縮官網關係：微縮官網提供右上角「品牌官網 / 身體觀察筆記」入口；品牌官網文章頁底部再導回北區 / 南區 / LINE / 電話。

- `index.html` 是手機導向的到店導航首頁。

- 「品牌官網 / 身體觀察筆記」應是右上角的站點切換入口，不應和主要到店 CTA 混在同一組。

## ObserveGeoPages 的內容來源

ObserveGeoPages 不是一般 SEO 文章站。它的內容來源包含：

- CaseAgent 去識別化案例
- CaseKnowledgeBase / 臨床決策邏輯庫
- 論文 / 專業資料庫
- 見觀品牌語言與法規風險邊界
- AdsControl 廣告策略與追蹤經驗
- ObserveSocialAutopilot 社群內容系統

其中「臨床決策邏輯庫 v3.1」是 CaseAgent 臨床判讀參考資料，強調輸入可信度、暫時性推論、紅旗分流、證據等級、保留式用語，以及「局部清場後殘留主線判讀」。它不是對外醫療承諾文件，不能直接公開成醫療結論。

## 公開內容安全邊界

ObserveGeoPages 可以使用醫療詞彙，但只能作為讀者常見問法、檢查名詞、搜尋入口或身體觀察起點。

可作為入口的詞包含：骨刺、手麻、腳麻、腰椎退化、神經壓迫、頸椎、腰椎、隧道症候群、梨狀肌。

禁止寫成：

- 見觀結構治療骨刺
- 改善神經壓迫
- 解決手麻腳麻
- 根治腰椎退化
- 保證不再疼痛
- 成功案例或前後對比
- 醫生沒告訴你的真相
- 不用開刀就能治好

應使用的語氣：

- 身體結構觀察
- 傳統整復推拿
- 姿勢與承重線索整理
- 張力與生活負荷觀察
- 整理身體線索
- 看懂身體狀態
- 較清楚的調整方向
- 可能、可先留意、暫時看起來、仍需觀察

## 關於我們技術的文案寫法

v3.1 更新後，專案內所有「關於我們如何看身體 / 如何整理技術線索」的文案，都應遵守以下轉譯規則。

可公開寫法：

- 先從主訴、動作、姿勢、承重、左右差與生活負荷整理線索
- 痛點是入口，不直接把痛點當成答案
- 會先看局部反應，再看是否仍有殘留線索指向更上游或下游的連動
- 會依當次動作、觸感、前後差異與回測逐步收斂方向
- 每次整理都只是當下較符合的暫時性推論，仍需持續觀察

避免寫成：

- 一看就知道根因
- 一摸就知道是哪裡造成
- 手的問題就是髖造成 / 膝蓋就是骨盆造成
- 根因已確認在某個遠端結構
- 局部有效就代表已經解決
- 遠端一鬆就證明真正問題只在遠端
- 幫你把錯位喬回去 / 歸位 / 校正完成

較適合用來描述技術的說法：

- 先把當下最明顯的症狀出口整理乾淨
- 再看殘留反應是不是顯示還有其他承重或張力主線
- 用回測與生活情境確認方向，而不是直接下結論
- 不是用單一病名解釋全部，而是把身體線索整理得更清楚

這些規則不只適用於文章，也適用於：

- `about/` 相關品牌介紹
- 未來 ObserveGeoPages 的技術介紹頁
- FAQ 與站內說明文字
- 對外合作、簡報、提案中的技術描述

## 文章母型

ObserveGeoPages 的文章應以這個母型為主：

1. 常見錯誤問法
2. 說明這個問法為什麼不完整
3. 去識別化生活情境
4. 作者視角：看到這些資訊會先看什麼
5. 多軸觀察推理：左右差、坐站走、承重、張力、生活負荷
6. 問法重構
7. 什麼情況不適合只觀察，需要分流
8. 收束到見觀結構定位

文章不是回答「這個病要怎麼治」，而是回答「拿到這個檢查名詞或身體感覺後，要怎麼把自己的狀態整理得更清楚」。

補充：第二批 Markdown 草稿目前是公開前的工作稿，結構比公開頁再更簡，實際轉成 HTML 時仍可依公開版需要補上更完整的導覽、CTA 或風險邊界說明。

## 未來修改注意事項

- 不要把 `observe888-visit-hub` 擴張成完整文章庫。
- 不要把 ObserveGeoPages 做成到店導航站。
- 不要沿用舊官網的模板邏輯或外包更新節奏。
- 微縮官網優先服務手機到店決策；品牌官網優先服務搜尋、理解與內容累積。
- 新增文章時要先安全化、去識別化、移除療效語言，再進入公開頁。
- AdsControl 可以引用 ObserveGeoPages 作為深層內容來源，但長文內容本體應由 ObserveGeoPages 承載。
